# oxtra

A Python library for autonomous multi-agent AI workflows.

You provide intent; oxtra drives it to completion. An Overseer (persistent LLM with read-only tools and structured memory) makes judgment calls and generates workflows. A Scheduler (deterministic event loop) validates, executes, and enforces. Agent steps (scoped LLM calls) do the actual work.

## Three components

- **Overseer** -- the brain. One persistent agent with read-only tools, PostgreSQL memory, and structured decision protocols. It observes everything, decides what should happen, generates workflows, and never does the work itself.
- **Scheduler** -- the nervous system. A deterministic event loop that validates workflows, executes steps, enforces budgets and constraints, classifies errors, and routes events. It has no opinions.
- **Agent steps** -- the hands. Scoped LLM calls inside workflows. Short-lived -- they finish within their context window and report back.

## Agent definitions

```toml
# agents/researcher.toml
[agent]
name = "researcher"
description = "Gathers information from web pages"
prompt = "researcher.md"
category = "quick"

[tools]
allow = ["read", "list_dir", "grep", "glob", "http", "consult", "notepad"]
```

Agents are static data documents (TOML metadata + markdown prompt), not code. The `allow` whitelist controls which tools the agent can use. Tools not listed are mechanically absent from the LLM's perspective. Prompts support `{include:filename.md}` for composition from multiple files.

## Workflows

```toml
[workflow]
name = "process-data"
description = "Research, generate, review"

[[steps]]
name = "research"
agent = "researcher"
task = "Investigate {target}."
variables = ["target"]
depends_on_previous = false
timeout = 300
verify = ["myproject.verify:research_complete"]

[[steps]]
name = "generate"
agent = "generator"
category = "deep"
task = "Generate output for {target}."
variables = ["target", "output_path"]
depends_on = ["research"]
timeout = 600
verify = ["myproject.verify:output_valid"]
verify_agent = "code-reviewer"
verify_block_threshold = "major"
```

Workflows are generated by the Overseer, not hand-written. They declare steps with dependencies, timeouts, verification chains, and retry policy. The scheduler builds a dependency graph and runs independent steps in parallel. Steps can be agent-driven (LLM), function-driven (Python callable), decision points (Overseer decides at runtime), gates (event waits), or child workflow spawns.

## Tools

```python
from oxtra.tool import Tool

async def search(args: dict) -> str:
    query = args["query"]
    return f"Results for: {query}"

search_tool = Tool(
    name="search",
    description="Search for information",
    parameters={
        "type": "object",
        "properties": {"query": {"type": "string"}},
        "required": ["query"],
    },
    execute=search,
)
```

Tools are single objects that bundle schema and implementation. oxtra provides constructors for common tools (`make_read_tool`, `make_write_tool`, `make_edit_tool`, `make_git_tool`, `make_exec_tool`, `make_http_tool`, etc.) but none are mandatory. No bash tool -- granular purpose-built tools with typed parameters and mechanical scoping instead.

## Categories

```toml
# categories.toml
[categories]
quick = "anthropic/claude-haiku-4-5"
standard = "anthropic/claude-sonnet-4-6"
deep = "anthropic/claude-opus-4-6"
```

A flat map from intent to model. No fallback chains, no fuzzy matching. Missing category is a hard error.

## Design principles

- **Agents are data, not code.** TOML + markdown. No factory functions, no classes, no lifecycle methods.
- **Permissions are whitelists.** The LLM never sees unlisted tools. Mechanically enforced, not prompt-instructed.
- **Subagents cannot delegate.** `spawn` is mechanically stripped from spawned agents. Workers can `consult` read-only agents for research.
- **Verification is mechanical.** Ordered chains of callables (cheapest first, short-circuit on failure), then verification agents returning structured verdicts with severity-gated blocking.
- **Structured decisions, not free-form.** The Overseer picks from menus via typed protocols. If a situation doesn't match any protocol, it escalates to the human.
- **No implicit defaults.** Provider, model, database, timeout, retry behavior -- all must be explicit.
- **No bash tool.** Granular tools (read, write, edit, git, exec, http) with typed parameters and path enforcement.
- **PostgreSQL backbone.** All state in PG. Append-only immutable tables. LISTEN/NOTIFY for cross-process observation. Advisory locks for mutual exclusion.

## Architecture

Three components, thirteen modules:

| Module | Purpose |
|---|---|
| `overseer/` | Persistent LLM, decisions, memory, learning, session handoff |
| `scheduler/` | Event loop, validation, execution, budgets, constraints, crash recovery |
| `agent/` | Load and validate agent definitions |
| `tool/` | Tool contract, registry, constructors (no bash) |
| `transport/` | LLM communication via Provider protocol (raw httpx) |
| `verify/` | Mechanical chains + semantic verification agents (structured verdicts) |
| `notepad/` | Cross-agent context sharing (PG-backed) |
| `session/` | Session lifecycle, token tracking |
| `trace/` | PG schema owner: events, results, transcripts, inbox |
| `knowledge/` | Knowledge ingestion and retrieval via cognee (PG+pgvector) |
| `services/` | Shared business logic for all frontends |
| `cli/` | strictcli CLI (agents are the primary users) |
| `mcp/` | MCP server (human interface via dashboard/AI client) |

See `DESIGN.md` for the full architecture, design axioms, and anti-patterns.
