# Scheduler Module Design

The scheduler is the nervous system. A deterministic event loop that receives workflows from the Overseer, validates them, executes them, enforces budgets and constraints, routes events, and manages state. It has no opinions. It does what it is told.

## Responsibility

- Validate and execute workflows (TOML generated by the Overseer)
- Sequence steps within workflows based on dependency graphs
- Route events to the Overseer (step completed, step failed, budget crossed, human responded)
- Enforce budgets (USD-based, per-workflow)
- Enforce mechanical constraints after each step
- Invoke the Overseer's decision protocols when events require judgment
- Spawn sanity-check subagents after Overseer decisions
- Track Overseer health metrics and enter degraded mode when needed
- Manage crash recovery via the trace module
- Manage write safety (per-path write queue, stale-write detection, transient replay)

## Workflow Validation

Before executing a workflow produced by the Overseer, the scheduler validates it:

1. **Schema validation**: rejects workflows with unknown step types, invalid state references, unreachable steps, impossible permissions, missing required fields, mutual exclusivity violations.
2. **Sanity-check subagent**: a cheap independent LLM call reviews: "Do file scopes match the task? Do step types make sense? Are dependencies consistent? Is budget plausible?"

If validation fails, the scheduler requests regeneration from the Overseer with the validation errors. Second failure escalates to the human.

## Workflow Definition Format (TOML)

Workflows are generated by the Overseer, not hand-written. The format:

```toml
[workflow]
name = "process-data"
description = "Full processing workflow: research, generate, review"

[[steps]]
name = "research"
agent = "researcher"
task = "Investigate {target}: gather relevant pages and extract key content."
variables = ["target", "work_dir"]
depends_on_previous = false
timeout = 300
verify = ["myproject.verify:research_complete"]
```

## Step Schema

Each `[[steps]]` entry has these fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | yes | Unique step identifier |
| `agent` | string | yes* | Agent name. Mutually exclusive with `callable`. |
| `task` | string | yes* | Task prompt. `{variable}` placeholders substituted from runtime variables. Mutually exclusive with `callable`. |
| `callable` | string | yes* | Python callable path (e.g., `"myproject.steps:fetch_data"`). Mutually exclusive with `agent` + `task`. |
| `variables` | array of strings | yes | Variable names this step needs. Hard error if any are missing at runtime. |
| `depends_on` | array of strings | yes** | Step names that must complete first. Mutually exclusive with `depends_on_previous`. |
| `depends_on_previous` | boolean | yes** | If true, depends on the immediately preceding step. Mutually exclusive with `depends_on`. |
| `timeout` | integer | yes (agent steps) | Max wall-clock seconds. No default. Not applicable to function steps. |
| `category` | string | no | Override the agent's default category. Agent steps only. |
| `retry` | integer | no | Max retry count. Default: 0. |
| `retry_resume` | boolean | conditional | Required if `retry > 0`. True = resume session. False = fresh session. |
| `retry_inject_failure` | boolean | no | Inject full structured failure context into retry prompt. Default: false. |
| `verify` | array of strings | no | Ordered list of Python callable paths for mechanical verification. Run in order, short-circuit on first failure. |
| `verify_agent` | string | no | Agent name for semantic verification (full agent definition, structured verdict). |
| `verify_block_threshold` | string | conditional | Required if `verify_agent` is set. One of `critical`, `major`, `minor`, `nit`. |
| `for_each` | string | no | Variable name resolving to a list. Step executes once per item. |
| `for_each_abort_on_failure` | boolean | conditional | Required if `for_each` is set. |
| `output_schema` | string | no | JSON Schema path for structured output validation. Agent steps only. |
| `budget` | number | no | Per-step USD budget. |
| `workflow` | string | no | Workflow name to spawn as a child. Mutually exclusive with `agent`/`task`/`callable`/`decision_point`/`gate`. |
| `wait` | boolean | conditional | Required if `workflow` is set. True = block until child completes. False = fire-and-forget. |
| `decision_point` | boolean | no | Pauses execution and invokes the Overseer. Mutually exclusive with other step types. |
| `gate` | string | no | Event name to wait for. Blocks until the named event fires or `timeout` expires. |
| `write_paths` | array of strings | no | File paths this step may write. When set, the scheduler re-constructs write/edit tools with `scope` restricted to these paths. |
| `on_success` | string | no | Python callable path. Runs after verification passes. Non-fatal. |
| `pre_retry` | string | no | Python callable path. Runs before a retry attempt for state cleanup. |

*A step must declare exactly one of: `agent` + `task`, `callable`, `workflow`, `decision_point`, or `gate`.

**Every step must declare either `depends_on` or `depends_on_previous`. Both missing or both present is a hard error.

## State Machine Enforcement

Step statuses and their legal transitions are enforced by the trace module. Invalid transitions are hard errors. Every transition emits a trace event with `{old_status, new_status, reason}`. See `trace/DESIGN.md` for the full state machine definition.

## Agent Step Context Assembly

Before each agent step, the scheduler builds the agent's context in three layers, then the Overseer refines it.

**Layer 1: Task declaration.** The task prompt from the workflow TOML, with `{variable}` placeholders substituted.

**Layer 2: Runtime system context.** Mechanically appended by the scheduler:
- Active constraints from the Overseer's memory (both mechanical and advisory)
- This step's prior failed attempts with full structured failure context (if retrying with `retry_inject_failure`)
- Notepad content from the current run

**Layer 3: Overseer-selected lessons.** The scheduler invokes the Overseer's `context_decision` protocol with the assembled context from layers 1-2, the step type, and a task summary. The Overseer can: select relevant lessons, request additional code context, reorder or trim any layer, or leave the context as-is.

The scheduler stores both versions (pre-refinement and post-refinement) via the trace module's `context_diffs` table.

### Consumer Context

Two mechanisms for injecting domain-specific content:

**Workflow variables.** Pre-computed context sections passed as variables and referenced in agent prompt templates via `{variable}` placeholders.

**Prompt file composition.** Agent prompt `.md` files support `{include:filename.md}`. See `agent/DESIGN.md`.

Both compose with the three-layer context assembly.

## Event Loop

The scheduler is an event-driven loop:

1. **Receive workflow** from Overseer -> validate -> create in trace
2. **Execute ready steps** (dependencies met, budget available) -> spawn agent or run callable
3. **Step completes** -> run verification chain -> run mechanical constraints -> record result
4. **Step fails** -> invoke Overseer's `retry_decision` protocol -> act on response
5. **Budget threshold crossed** -> invoke Overseer's `budget_decision` protocol -> act on response
6. **Human responds to inbox item** -> update assumption status -> fire answer event (for gates) -> invoke Overseer if response contradicts assumption
7. **All steps complete** -> workflow done -> notify Overseer

Between steps, the scheduler checks:
- Mechanical constraints (from the Overseer's `constraint_decision` outputs)
- Budget limits (USD)
- Timeout enforcement
- Crash recovery checkpoint (state persisted via trace)

## Dependencies and Parallelism

The scheduler builds a dependency graph from `depends_on` and `depends_on_previous` declarations. Steps with no dependency between them run in parallel via asyncio. `for_each` iterations are also parallel.

Topological sort at load time. Cycles are hard errors.

## Per-Workflow Budgets

Each workflow has its own USD budget (set by the Overseer). A failing workflow burns its own allocation. When a workflow's accumulated cost approaches its budget, the scheduler invokes the Overseer's `budget_decision` protocol. The Overseer can reallocate from completed workflows, increase allocation, or abort.

Cost is computed from token counts using oxtra's internal pricing table.

## Mechanical Constraint Enforcement

After each step's verification chain, the scheduler checks all active mechanical constraints (from the constraints table). Violations are immediate step failures.

## Function Steps

A function step runs a Python callable instead of an LLM agent. `callable` field, `StepContext` input, `StepResult` output. Variable name collisions are hard errors. Function steps do not support `category`, `timeout`, or `verify_agent`.

## Structured Output

Steps with `output_schema` get JSON extraction and validation after completion. Parse failure or validation failure triggers retry. Validated output available as `{step_name}_output`.

## Batch Iteration

Steps with `for_each` execute once per item in a list variable. Iterations run in parallel. `for_each_abort_on_failure` controls whether partial failures abort. `for_each` + `output_schema` validates per iteration, collects as list.

## Post-Step Actions

**`on_success` callable.** Runs after verification passes. Receives a `StepContext`. Non-zero return or raised exception is logged but does not fail the step.

**`pre_retry` callable.** Runs before a retry attempt. For cleaning up state: reverting broken files, resetting environment, clearing caches.

## Dynamic Fan-Out from Agent Output

Compose `output_schema` and `for_each` across two steps. The planner's validated output becomes the `for_each` input for the worker step. `{item}` provides dot-notation access to fields.

## Decision Point Steps

Pauses workflow execution and invokes the Overseer. The Overseer sees the current workflow state and decides: continue as-is, modify remaining steps, or abort.

## Gate Steps

Blocks until a named event fires or the timeout expires. Useful for inter-workflow synchronization, waiting on external events, or awaiting an inbox item's answer (via the `answer_event` field on inbox items, delivered through PG LISTEN/NOTIFY).

If the event fires before timeout, the step succeeds and the event payload is available as `{step_name}_event`. If timeout expires, the step fails.

## Services

Workflows can declare long-running processes with start/health-check/stop commands. The scheduler manages their lifecycle: starts before the first step that needs them, monitors health, stops on workflow completion or crash. Runtime details (port, URL, PID) accessible via variable substitution.

## File-Lock Registry

When multiple workflows run concurrently, the file-lock registry prevents conflicting file writes. File-level granularity.

A workflow claims files at creation (inferred from `write_paths` declarations). Claims are all-or-nothing. If an agent step discovers it needs a file outside its workflow's claims, the scheduler escalates to the Overseer's `scope_decision` protocol.

Claims are released when the workflow completes, fails, or is aborted.

## Cross-Workflow Data Flow

One workflow can declare read-only access to another workflow's state fields:

```toml
[workflow]
name = "generate-report"
reads_from = {research_workflow = ["findings", "sources"]}
```

The scheduler resolves these at step execution time. If the source hasn't produced the field yet, the step blocks until available (with timeout). Values are read-only copies.

## Step Output Propagation

When step B depends on step A, B has access to A's outputs:

- **Structured output**: `{step_a_output}` (validated JSON)
- **Text output**: `{step_a_text}` (raw agent response)
- **Result metadata**: `{step_a_result}` with `{passed, duration_seconds, retries_used}`

Variable name collisions between step outputs and workflow variables are hard errors.

## Cancellation

- **asyncio cancellation**: catches `CancelledError`, cleans up sessions, writes partial results
- **`abort()` method**: sets internal flag, triggers cancellation

## Crash Recovery

Three-pass idempotent startup recovery (see `trace/DESIGN.md` for details):

1. Reclaim interrupted steps (transition to `failed`)
2. Re-evaluate blocked/gated work
3. Clean orphaned resources (services, locks, stale advisory locks)

## Pause/Resume

Pause persists cursor + state via trace. Resume reloads and restores. Pauseable at every level (step, workflow, entire run).

## Write Safety Enforcement

The scheduler manages the four write-safety mechanisms on behalf of all file-mutating tools:

- Instantiates the per-path write queue at run start
- Passes the queue reference to tool constructors
- Re-constructs scoped tools per step when `write_paths` is declared
- Manages stale-write detection state per agent session

## Files

| File | Contents |
|---|---|
| `_types.py` | `StepContext`, `StepResult`, `WorkflowConfig` (parsed TOML), `WorkflowState`, `ServiceConfig`. Step schema field definitions. All pydantic models with `strict=True, extra='forbid'`. |
| `_loader.py` | `load_workflow(path_or_str)` -- reads workflow TOML, validates schema via pydantic, returns `WorkflowConfig`. |
| `_graph.py` | Dependency graph construction. Topological sort. Cycle detection. Parallelizable step identification. |
| `_executor.py` | `Scheduler` class. Event loop, step execution, timeout enforcement, retry logic, verification dispatch, constraint enforcement, budget tracking, pause/resume, `abort()`. Write safety orchestration. |
| `_validator.py` | Workflow validation: schema checks + sanity-check subagent dispatch. |
| `_services.py` | Service lifecycle management. Start, health-check, stop. Runtime detail injection. |
| `_locks.py` | File-lock registry. Claim management, conflict detection, release on workflow completion. |
| `_events.py` | Event registry for gate steps. Event registration, firing, listener management. Integrates with PG LISTEN/NOTIFY for inbox answer events. |

## What This Module Does NOT Do

- Does not make judgment calls (that's the Overseer)
- Does not generate workflows (that's the Overseer)
- Does not define agents or tools (those are loaded externally)
- Does not manage transport connections (that's session/)
- Does not implement model selection logic (that's agent/ category resolution)
- Does not decide retry strategy (asks the Overseer via protocol)
- Does not compute USD costs (reads the pricing table; computation is mechanical)
